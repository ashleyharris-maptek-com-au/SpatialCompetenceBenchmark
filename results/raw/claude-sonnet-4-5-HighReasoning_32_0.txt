# File Format Optimization Report: 3D Laser Scan Data

## Executive Summary

This report analyzes optimization strategies for a laser scanning file format handling hundreds of millions to billions of beam measurements. The proposed format must balance extreme compression with query performance while respecting hardware precision limits of 20-bit angles, 12-bit signal strength, and 16-bit timing resolution.

## Core Optimization Strategies

### 1. **Exploit Scanner Geometry (STRONGLY RECOMMEND)**

The predictable sweep pattern is our greatest asset. The vertical mirror at 600rpm creates a regular raster pattern. Rather than storing absolute angles for every beam:

- **Delta encoding**: Store the first beam's full position (20 bits × 2), then encode subsequent beams as deltas. Within a single sweep, elevation changes predictably; deltas may require only 8-12 bits.
- **Sweep reconstruction**: Store sweep metadata (start angle, sweep rate, timestamp) and beam index within sweep. The format can mathematically reconstruct angles, reducing storage from 40 bits per beam to ~4 bytes per sweep header plus 2-3 bytes per beam index.
- **Compression ratio**: ~85% reduction in angular data

**Justification**: This approach is lossless within the 20-bit encoder resolution and leverages the fundamental physics of the device.

### 2. **Variable Return Count Encoding (STRONGLY RECOMMEND)**

The number of returns varies dramatically:

- Use 2-bit prefix to indicate return count: 00=no return (skip remaining data), 01=single return, 10=2-3 returns (1 more bit), 11=4+ returns (read additional byte)
- Most beams return 0-1 hits; this front-loads the common case
- For no-return beams (atmosphere shots), only 2 bits are needed versus 40+ bits for full data

**Compression ratio**: 95%+ for no-return beams, which may constitute 20-40% of outdoor scans

### 3. **Separate Hot and Cold Data (STRONGLY RECOMMEND)**

Partition the file into multiple streams:

- **Geometry stream**: Beam positions and return counts (frequently accessed)
- **Intensity stream**: Signal strength values (12-bit, frequently accessed)
- **Image stream**: RGBA data (occasionally accessed)
- **Metadata stream**: Scanner position, timestamps, calibration (rarely accessed)

**Justification**: Users typically query geometry first, then selectively load intensity/color. This reduces I/O for typical workflows by 60-80%.

### 4. **Spatial Chunking with R-tree Index (STRONGLY RECOMMEND)**

Divide scans into 3D spatial chunks (e.g., 10m × 10m × 10m):

- Build an R-tree index mapping bounding boxes to file offsets
- Enables spatial queries without scanning entire files
- Chunk size should target 1-10MB for modern SSD optimization

**Justification**: Billion-beam files are 50-200GB; spatial indexing makes them practically usable. 99% of queries are spatially local.

### 5. **Image Data Optimization (STRONGLY RECOMMEND)**

RGBA data at 1+ pixels per 2×2 beams means ~1 pixel per beam:

- Store images as standard compressed formats (JPEG-XL for lossy, PNG for lossless) within chunks
- Include mapping table linking beam coordinates to pixel coordinates
- 4K resolution at 8-bit RGBA = 64MB uncompressed but 2-10MB compressed

**Compression ratio**: 90-97% using existing image codecs

**Justification**: Don't reinvent image compression; leverage hardware-accelerated decoders.

### 6. **Bit-Packed Fixed-Width Fields (RECOMMEND)**

Never use byte-aligned fields when data doesn't require it:

- 20-bit angles (not 24-bit/3-byte)
- 12-bit intensity (not 16-bit)
- Custom bit-stream writer/reader

**Compression ratio**: 25-33% savings over byte-alignment

**Caveat**: Slightly slower parsing but worthwhile for hundreds of gigabytes.

### 7. **General Purpose Compression Layer (RECOMMEND WITH CAUTION)**

Apply LZ4/Zstandard to chunks after domain-specific encoding:

- **LZ4**: 2-3× compression, extremely fast decompression (>1GB/s)
- **Zstandard**: 4-6× compression at level 3, still fast (~400MB/s)
- Delta-encoded data compresses exceptionally well

**Justification**: After domain encoding, data still has patterns. But adds complexity and CPU overhead.

**Recommendation**: Make optional; users choose speed vs. size tradeoff.

## Unwise But Interesting Optimizations

### 8. **Lossy Coordinate Quantization (NOT RECOMMENDED)**

Reduce 20-bit angles to 16-bit or less.

**Why tempting**: 20% storage reduction
**Why unwise**: Exceeds A/D converter precision limits (explicit requirement violation). At 100m range, losing 4 bits degrades accuracy from ~1cm to ~16cm—unacceptable for infrastructure surveying.

### 9. **Machine Learning Prediction Models (NOT RECOMMENDED)**

Train neural networks to predict return intensities based on geometry.

**Why tempting**: Could achieve 10× compression
**Why unwise**: 
- Adds model versioning complexity
- Inference overhead on decompression
- "Lossy in unpredictable ways" violates immutability principle
- Hallucinated data in professional surveying is legally problematic

### 10. **Octree/Point Cloud Decimation (NOT RECOMMENDED FOR PRIMARY FORMAT)**

Store hierarchical LOD using octree subdivision.

**Why tempting**: Enables fast visualization
**Why unwise as primary format**: Destroys original beam structure and temporal information. However, should be supported as secondary index structure.

### 11. **Floating-Point XYZ Conversion (NOT RECOMMENDED)**

Convert spherical coordinates to Cartesian XYZ in 32-bit float.

**Why tempting**: Standard format for most 3D software
**Why unwise**: 
- 32-bit float has ~7 decimal digits precision
- At 1000m range, precision degrades to ~10cm
- 20-bit angular encoding gives ~1-2cm at same range
- Storage increases from 40 bits (angles) to 96 bits (xyz)

**Recommendation**: Support export to XYZ but don't use as native storage.

## GPS/Position Data Handling

### 12. **Differential GPS Encoding (RECOMMEND)**

For moving platform scans:
- Store initial GPS position at full precision (64-bit double × 3)
- Use local coordinate frame (offset in meters as 32-bit float)
- Update origin only when offset exceeds ±1km

For static scans:
- Single GPS position in header
- No per-beam position data

**Compression ratio**: 99.9% for static, 99%+ for moving scans

## Time-Series Encoding

### 13. **Temporal Delta Encoding (STRONGLY RECOMMEND)**

Store base timestamp per chunk (64-bit Unix nano), then:
- Delta encode beam times as 16-bit values (65ms range at 1μs resolution)
- At 600rpm vertical rotation, full sweep = 100ms, so 16 bits covers multiple sweeps
- Overflow to new chunk if needed

**Compression ratio**: 75% reduction in timestamp data

## File Structure Recommendation

```
[HEADER]
- Magic bytes: "LSCAN01"
- Coordinate system, scanner calibration
- Global bounding box
- Chunk count, R-tree index offset

[CHUNKS] (spatially organized)
  Chunk header:
  - Bounding box
  - Base timestamp, GPS position
  - Stream offsets (geometry, intensity, image)
  
  Geometry stream (Zstandard compressed):
  - Sweep headers (angular rates, base angles)
  - Per-beam: 2-bit return count + delta angles (3-5 bytes avg)
  
  Intensity stream (Zstandard compressed):
  - Bit-packed 12-bit values
  
  Image stream:
  - JPEG-XL compressed RGBA
  - Beam-to-pixel mapping table

[R-TREE INDEX]
- Binary tree structure for spatial queries

[METADATA]
- Processing notes, operator info
```

## Conclusion

The recommended optimizations exploit domain-specific knowledge (scanner geometry, spatial coherence, typical query patterns) while staying within precision requirements. Expected compression versus naive storage:

- **Naïve**: 40 (angles) + 12 (intensity) + 32 (RGBA) = 84 bits/beam = 10.5 bytes/beam
- **Optimized**: ~2-4 bytes/beam average after domain encoding + compression

**Final compression ratio: 60-80%**, meaning a billion-beam scan compresses from 10GB to 2-4GB, with spatial indexing enabling sub-second queries. This provides the competitive edge through both storage efficiency and operational performance.
