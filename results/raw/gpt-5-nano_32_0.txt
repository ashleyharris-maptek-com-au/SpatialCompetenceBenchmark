Executive summary
- Build a compact, immutable, binary container designed around the physics of the scanner (two rotating mirrors, beams with variable returns, image data tied to beam cells) and the need for large-scale scans (hundreds of millions to hundreds of billions of beam records).
- Favor block/chunked, columnar-like storage with heavy use of delta/difference coding, quantization (lossy where acceptable), and bit-packing to minimize overhead. Separate “geometry/pose” data, per-frame framing, per-beam data, and color image data to maximize compressibility and enable selective loading.
- Adopt a data layout that enables streaming writes (append-only), fast progressive decoding, efficient random access (by time, azimuth, or spatial region), and optional lossless or lossy compression modes.
- Acknowledge unwise but commonly tempting choices (e.g., naive JSON blobs, per-beam fixed-width fields with no compression, ad-hoc custom headers without versioning) and explain why they hurt scalability, interoperability, or performance at scale.

1) Goals and constraints
- Immutable file: can be written incrementally but not modified after closing. Index/footers can be appended at the end or included as a final pass.
- Scale: hundreds of millions to hundreds of billions of beam records; color image data tied to 2x2 beam blocks; GPS-based origin/orientation for each frame; variability in number of returns per beam.
- Data channels and resolutions:
  - Range (distance) per return
  - Intensity (12-bit)
  - Time per return (16-bit)
  - Azimuth and elevation per beam (20-bit per axis, but you can derive these in many layouts; not all per-beam storage is strictly necessary if you encode geometry by frame and index)
  - Surface descriptors (albedo, normal) per hit or per beam return (requires careful quantization)
  - Multi-return handling (unknown number per beam)
  - Image data: RGBA per 2x2 beam block, with 1 or more pixels per block
  - Pose data: GPS/position, frame orientation; possible vehicle motion
- Acceptable data loss: lossy quantization/delta encodings within the stated bit budgets (12-bit intensity, 16-bit time, 20-bit angles) to maximize compression without unacceptable degradation.

2) High-level architecture: a compact, chunked container
- Container format: a binary file with a small fixed header, a stream of framed segments (frames), and an index/footer at the end.
- Frame definition: a logical time slice or frame corresponding to a slow-rotation step (azimuth) plus a vertical sweep (elevation) grid. Each frame contains:
  - Frame header: frame_id, frame_start_time (absolute or relative), azimuth_offset (frame-wide), number_of_vertical_steps, vertical_step_angle, number_of_beams_in_frame
  - Optional per-frame pose: GPS time, position (x,y,z), orientation quaternion or yaw/pitch/roll; frame-local pose for later global reconstruction
  - Per-beam data block (see below)
  - Optional per-frame color/image tile reference
- Beam data organization options:
  - Option A (preferred for memory locality and compression): per-frame, per-beam records stored contiguously, but with:
    - Angles not stored per-beam; instead, frames carry the azimuth offset and vertical grid indices; absolute angles are reconstructed from indices and frame geometry
    - R uses delta-encoded or delta-encoded-like storage depending on the data
  - Option B: store per-beam absolute angles (20 bits each). This is straightforward but less compressible; use if you want simpler tooling and are not constrained by bandwidth.
- Image data layout:
  - Color data stored as a separate image tile per frame or per some frame group.
  - Color data uses 2x2 beam blocks mapped to one or more RGBA pixels.
  - Use a small palette-based or indexed representation where feasible; otherwise store compressed RGBA (PNG/WebP-like) per tile, with a simple header indicating compression method and tile dimensions.
- Index/metadata:
  - End-of-file footer containing an index:
    - Frame offsets, frame IDs, per-frame metadata (pose block offset, color tile offset)
    - Optional spatial/index grid mapping for rapid region queries (coarse 3D grid or octree indices)
  - Optional separate index blocks for time-based seeking (frame_start_time)

3) Data modeling: per-beam and per-return fields
- Core per-beam fields (if you store absolute per-beam angles):
  - frame_id or frame_index
  - azimuth_angle (20 bits)
  - elevation_angle (20 bits)
  - timestamp (16 bits) or delta time from frame_start
  - number_of_returns (8 bits, or 4 bits if you bound typical returns)
  - For each return j (j = 1..Nreturns):
    - range (distance) (32-bit unsigned fixed-point or 32-bit float, to maintain accuracy)
    - intensity (12 bits; store with 4 extra padding or pack two returns into 32-bit words)
    - return_time (16 bits)
    - albedo/normal data: consider quantized normals (e.g., 16-bit packed normals or 8 bits per axis) and a 12–16 bit reflectance payload
- Alternative per-beam approach (more compression-friendly):
  - Frame-wide angle grid provides elevation index and azimuth index for each beam; store:
    - frame_id, beam_index (implicit by order)
    - 16–20-bit delta-time
    - Nreturns
  - For each return:
    - range, intensity, return_time
    - Optional normal/albedo as quantized components
- Multi-return handling:
  - Store returns as a small list per beam:
    - A single per-beam “returns_count” indicator
    - Then for each return: a compact tuple as above
  - If the average number of returns is small (1–2) this is efficient; for dense multi-returns, you can use a compact encoding (delta of ranges, delta of times across returns, shared albedo/normal signature)

4) Encoding techniques to maximize compression (wise choices)
- Quantization and lossy reduction (where acceptable)
  - Elevation/azimuth indices: encode as fixed-width indices with optional frame offset; avoid per-beam 40-bit angle words
  - Range: store as scaled integer (e.g., 24-bit fixed-point) with controlled quantization error
  - Time: store 16-bit delta times between consecutive returns within a frame; if time drift is small, this can be highly compressible
  - Intensity: keep 12-bit resolution; pack into 16-bit words across multiple returns
  - Normals/albedo: quantize to 8–16 bits per component, or store a small normal index into a low-precision shared table
- Delta and predictive coding
  - Spatial/temporal deltas: store differences between successive beams or within a frame to exploit locality
  - Frame-to-frame pose and azimuth offsets are typically smooth; encode a single initial pose per frame and store only a few deltas
- Bit-packing and variable-length integers
  - Use base-128 varints or a custom small-bit-field packer for counts, small deltas, and indices
  - Pack small fields tightly (e.g., 12-bit intensity + 4 bits padding to 16 bits; 20-bit angles packed into 3 bytes plus 4 bits)
- Separate, optional lossless compression pass
  - After bit-packing, optionally run a general-purpose compressor (Zstandard, Zstd-LZMA, or Zlib) per frame or per segment
  - For color tiles, use standard image codecs (PNG for lossless, WebP for lossy) at tile granularity
- Frame- and frame-group level optimization
  - Store a frame header with a frame-level pose and a frame-wide azimuth offset; avoid per-beam angle storage
  - Group consecutive frames into larger segments if they share a common frame geometry or temporal continuity
- Image/color data optimization
  - Map 2x2 beam blocks to 1–4 color pixels; if most blocks have a single color, encode as a palette index with a small palette
  - If color variation is high, store raw compressed RGBA blocks per tile
  - Consider interleaving color tiles with geometry data to improve cache locality during decoding
- Metadata and schema
  - Use a clear, versioned schema (e.g., a header with a version field and a reusable, extensible metadata block)
  - Store units for all numeric fields (meters, radians, microseconds, etc.)
  - Use standard coordinate system references (EPSG, local frames) rather than ad hoc strings
  - Include a quality/flags field per frame (e.g., data-loss tolerance mode, whether frame uses pose as absolute origin)
- Indexing and fast access
  - End-of-file index that maps frame_id to byte offsets
  - Optional coarse spatial index (frame-space grid or octree-like) for rapid region queries
  - Time-based indexing to locate frames by GPS/posture timestamps
- Validation and integrity
  - Include a content checksum (CRC or XXH3/CRC64) per frame and per file footer
  - Optional digital signature for immutable provenance
- Streaming write optimization
  - Write frames sequentially; defer index/footer construction until close
  - Use a small, fixed frame header size to amortize overhead
  - Align to 4- or 8-byte boundaries for faster I/O

5) Data layout alternatives and trade-offs
- Absolute-angle per-beam storage (simple, straightforward):
  - Pros: easy to read/write; straightforward decoding
  - Cons: poor compression; large overhead for 20-bit angles per beam at scale
  - Recommendation: use absolute angles only if you need extreme simplicity or compatibility with existing codebases that require easy, per-beam angle access
- Frame-local geometry with derived angles (recommended):
  - Pros: dramatically better compression; angles derived from frame geometry and indices; excellent for streaming and random access by frame
  - Cons: requires a well-documented geometry model and careful calibration
  - Recommendation: adopt frame-based geometry with per-frame pose and per-grid indices
- Frame-based delta times and deltas for other fields:
  - Pros: high entropy reduction; excellent for long sequences
  - Cons: potential decoding complexity; slightly more complex error handling
  - Recommendation: include optional delta-encoded mode that can be selected via header flags
- Color data as image tiles vs. per-beam color:
  - Tile-based color: reduces per-beam overhead; easy to compress with image codecs
  - Per-beam color: straightforward but heavy data overhead
  - Recommendation: color tiles; provide a minimal palette option for blocks with uniform color
- Lossy vs. lossless modes:
  - Lossless: necessary for many survey tasks and legal requirements
  - Lossy: appropriate for “quality vs. size” tuning; provide deterministic quantization settings and a per-field loss policy
  - Recommendation: implement both as selectable modes, with default to lossless for measurements-critical work but include a calibrated lossy mode for very large datasets or preview workflows

6) Practical recommendations: what to adopt (and what to avoid)
Wise choices to adopt
- Frame-based geometry with indices, not per-beam angles
  - Reduces data by avoiding 40 bits per beam for angles and increases compressibility
- Bit-packed, delta-encoded per-frame data
  - Greatly reduces the average bits per return; keeps data within tight budgets
- Quantization with controllable loss
  - Provide explicit quantization parameters (e.g., range unit, angle unit, normal/reflectance precision)
  - Allow users to choose lossless, moderately lossy, or aggressive lossy modes
- Separate sections for geometry, per-beam returns, and color
  - Improves cache locality and allows independent evolution of components
- Image tiles for color data with a simple palette option
  - Enables standard image compression pathways and selective decoding
- Append-only design with a final index/footer
  - Supports streaming writes, easy integrity checks, and offline indexing
- Metadata with versioning, units, coordinate system, and calibration data
  - Essential for interoperability and future-proofing
- Strong integrity checks (checksums, optional signatures)
  - Ensures immutability and data provenance
- Optional coarse spatial/time index for fast querying
  - Beneficial for visualization and targeted analysis

Unwise choices to avoid (or do only with clear justification)
- Storing every angle per beam as 20-bit (or larger) fields without a frame-based derivation
  - Increases file size dramatically and reduces compressibility
- Heavy, unstructured JSON/XML metadata inside the file
  - Increases size and hinders streaming; prefer a compact binary metadata blob
- Repeated, redundant pose information per beam
  - Wasteful; instead store per-frame pose plus frame-derived angles
- Mixing disparate data schemas without a single versioned spec
  - Creates long-term interoperability problems; prefer a single, versioned schema with backward compatibility
- Proprietary, ad-hoc compression schemes without reference implementations
  - Inhibits adoption; lean toward standard compression layers (zstd, deflate) or well-supported image codecs for color
- Ignoring time synchronization between GPS/pose and beam timestamps
  - Leads to misalignment in downstream processing; ensure a clean, defined time reference for all frames
- Neglecting a robust index and fast query capabilities
  - For very large datasets, a sparsely indexed file will be painful to use; invest in a compact frame/time index
- Over-reliance on lossy encoding without user controls
  - If data must be reprocessed or validated later, lossy modes can hinder reproducibility; always provide a strict deterministic mode

7) A concrete, implementable design sketch (summary)
- File header (fixed size, versioned)
  - Magic/kind, version numbers, endianness flag
  - Global coordinate system reference (EPSG code or local frame ID)
  - Unit definitions (meters, degrees, time units)
  - Flag set: supports_frame_geometry, supports_color_tiles, lossy_mode, etc.
  - Global calibration data pointer or inline calibration block
- Frame block (variable length, sequential)
  - Frame header:
    - frame_id, frame_start_time (absolute or epoch), azimuth_offset (degrees), vertical_steps, vertical_angle_step
    - number_of_beams
    - frame_pose: position (x,y,z), orientation (quaternion or yaw/pitch/roll); optional IMU data
    - color_tile_ref: if color image is stored elsewhere in this frame
  - Beam data (for each beam in frame order)
    - If using derived angles: beam_id, delta_time (16-bit), Nreturns (8-bit)
    - If using absolute angles: azimuth_angle (20-bit), elevation_angle (20-bit), time (16-bit), Nreturns
    - For each return j:
      - range (32-bit fixed-point or float32)
      - intensity (12-bit)
      - return_time (16-bit)
      - normal/albedo (optional; quantized components)
  - Color tile (optional; per frame or per group)
    - tile width/height, tile data (palette index + RGBA or compressed image data)
- Index/footer
  - Frame table: frame_id -> offset, length
  - Optional coarse spatial index and time index
  - Validation checksum
- Compression options
  - On the wire (in the header or per-block flags): select:
    - LOSSLESS: no quantization; bit-packed fields only
    - LOSSY: define quantization parameters for range, time, angles, normals, and intensity
  - Post-processing compression: apply zstd/WebP/PNG on entire frame blocks or image tiles
- Extensibility
  - Reserve a version field; define a small extension mechanism for future fields (e.g., additional returns, new color channels, new sensor metadata)

8) Implementation considerations and testing
- Reference implementation
  - Build a small open-source reference reader/writer to lock down the spec, test round-trips, and measure compression ratios across representative datasets (airborne vs terrestrial scans, urban vs rural scenes)
- Benchmarks
  - Measure per-beam data size, per-frame data size, and total file size for realistic scanning rates
  - Compare with baseline formats (e.g., LAS/LAZ) on the same data, both in lossless and lossy modes
- Validation
  - Ensure that angle derivation from frame geometry yields the same point cloud (within quantization error) as per-beam absolute-angle mode
  - Validate multi-return ordering and times
- Tools
  - Provide a set of command-line utilities for: validating, converting from existing scans, sampling frames, and visualizing both geometry and the color tiles
- Interoperability
  - Document coordinate systems, units, and a mapping to and from LAS/LAZ where practical
  - Provide a compatibility layer to import/export to standard formats for compatibility with existing pipelines

9) Conclusion: recommended path
- Adopt a frame-based, derived-geometry data model with per-frame pose and per-grid-angle indices, not per-beam absolute angles.
- Use a bit-packed, delta-encoded core for range, time, and intensity, with 12–16 bit fields and careful packing to minimize waste.
- Include a color tile system with optional palette compression to handle the 2x2-beam color data efficiently.
- Provide both lossless and configurable lossy modes, with explicit quantization controls so users can trade quality for file size as needed.
- Implement a robust end-of-file index for fast random access and progressive loading, plus a footer checksum for integrity.
- Separate metadata, geometry, and image data into distinct sections to maximize compressibility and modularity.
- Avoid early adoption of bespoke, unstandardized compression schemes; rely on proven bit-packing, delta encoding, and standard compression layers (zstd, PNG/WebP) for the rest.
- Favor a design that scales to the data volumes you described, supports streaming writes and post hoc indexing, and remains interoperable with existing LiDAR data ecosystems.
